Defining and calling functions (cont.)

Long form for more complex functions:

function axpy!(y, a, x)
    for idx in eachindex(x, y)
        y[idx] = muladd(a, x[idx], y[idx])
    end
    return y
end

In a nutshell (cont.)

Type annotations in signatures are only used for dispatch, not performance. Multiple dispatch enables composability:

Rock, paper, scissors

abstract type Shape end
struct Rock     <: Shape end
struct Paper    <: Shape end
struct Scissors <: Shape end
play(::Paper,   ::Rock)     = "Paper wins"
play(::Paper,   ::Scissors) = "Scissors wins"
play(::Rock,    ::Scissors) = "Rock wins"
play(::T,       ::T) where {T<: Shape} = "Tie, try again"
play(a::Shape, b::Shape) = play(b, a) # Commutativity


play(Paper(), Scissors())
"Scissors wins"
play(Rock(), Rock())
"Tie, try again"
play(Rock(), Paper())
"Paper wins"

f(x) = 3 * sin(x) * cos(x) - 5 * exp(-x) + 2

Benchmarking and profiling

The @time macro

Julia comes with a simple macro @time for measuring elapsed time of sufficiently long-running functions:

@time sleep(0.1)
  0.101148 seconds (6 allocations: 224 bytes)

This may not provide useful information for very quick functions (resolution is of the order of ~millisecond):

using LinearAlgebra
A = randn(100)
norm(A) # warm up
@time norm(A)
  0.000004 seconds (1 allocation: 16 bytes)
9.147004157678

Using external profilers

Julia is also compatible with third-party profilers:

  • Linux perf, also via LinuxPerf.jl
  • Intel VTune (on x86-64), also for MPI programs, IntelITT.jl for instrumentation
  • NVIDIA Nsight Systems (on x86-64 and aarch64), also for MPI and GPU programs, NVTX.jl for instrumentation
  • Tracy
  • Other profilers that have been used: LIKWID.jl, Extrae.jl, ScoreP.jl, and more

Performance tips

General advices for improving performance of Julia code:

  • avoid accessing (untyped) global variables: put code in functions, don’t work in global scope
  • Julia has a garbage collector (GC) for safe automatic memory management, but in some cases it can get in the way of performance: do fully in-place operations in hot loops to avoid the GC to kick in and achieve best performance
  • the compiler may not always be able to prove array indexing operations are in bounds: you may want to use @inbounds to forcibly disable bounds checking (use with caution!). Double check this is actually necessary with @code_llvm, and use generic abstracts (like eachindex) whenever possible
  • avoid type-instabilities

function tmap(fn, itr)
    # for each i ∈ itr, spawn a task to compute fn(i)
    tasks = map(i -> @spawn(fn(i)), itr)
    # fetch and return all the results
    return fetch.(tasks)
end

Multi-threaded for loops

using ChunkSplitters, Base.Threads, BenchmarkTools

function sum_threads(fn, data; nchunks=nthreads())
    psums = zeros(eltype(data), nchunks)
    @threads for (c, elements) in enumerate(chunks(data; n=nchunks))
        psums[c] = sum(fn, elements)
    end
    return sum(psums)
end

v = randn(20_000_000);

@btime sum(sin, $v);
